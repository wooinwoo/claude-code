# Pull Request 작성 가이드

## 핵심 원칙

1. **간결함** - 필요한 정보만, 군더더기 없이
2. **명확함** - 리뷰어가 맥락 없이도 이해 가능
3. **일관성** - 동일한 형식으로 작성

## PR 제목 규칙

```
<type>(<scope>): <description> <JIRA-KEY>
```

- 70자 이내
- 영문 사용 권장 (한글도 허용)
- 동사 원형으로 시작
- **브랜치명에 Jira 키가 있으면 제목 맨 뒤에 추가 (괄호 없이)**

**Jira 키 패턴:** `[A-Z]+-[0-9]+` (예: GIFCA-123, DEV-456)

**예시:**
- 브랜치 `GIFCA-123-voucher-feature` → `feat(gifca/app): add voucher expiration notification GIFCA-123`
- 브랜치 `DEV-456-fix-payment` → `fix(gifca/core): resolve payment callback race condition DEV-456`
- 브랜치 `feature/add-auth` → `refactor(packages/drizzle): simplify transaction handling` (Jira 키 없음)

## PR 본문 템플릿

### 기본 템플릿

```markdown
## 개요
{이 PR이 해결하는 문제 또는 추가하는 기능}

## 주요 변경사항
- {논리적 단위로 정리된 변경사항}

## 테스트
- [ ] E2E 테스트 추가/수정
- [ ] 로컬 테스트 완료
- [ ] 기존 테스트 통과

## 참고사항
{필요한 경우에만 - 리뷰 포인트, 제약사항, 후속 작업 등}
```

### 버그 수정 템플릿

```markdown
## 개요
{어떤 버그를 수정하는지}

## 원인
{버그 발생 원인}

## 해결 방법
{어떻게 수정했는지}

## 테스트
- [ ] 버그 재현 테스트 추가
- [ ] 기존 테스트 통과
```

## 작성 시 주의사항

### 해야 할 것
- 변경의 "왜"를 설명
- 커밋들을 논리적 단위로 요약
- 리뷰어 관점에서 필요한 컨텍스트 제공
- 영향 범위가 넓으면 명시

### 하지 말아야 할 것
- 커밋 메시지 그대로 나열
- 코드 변경사항 그대로 복사
- 불필요한 섹션 포함 (스크린샷이 필요없으면 제외)
- 자명한 내용 반복 ("이 PR은 ~를 합니다" 같은 표현)

## 선택적 섹션

상황에 따라 추가:

### Breaking Changes (있을 경우만)
```markdown
## Breaking Changes
- {기존 동작이 변경되는 내용}
- {마이그레이션 방법}
```

### 스크린샷 (UI 변경시만)
```markdown
## 스크린샷
| Before | After |
|--------|-------|
| img    | img   |
```

### 관련 이슈 (있을 경우만)
```markdown
## 관련 이슈
- Closes #123
- Related: GIFCA-456
```

## 체크리스트

PR 생성 전 확인:
- [ ] 제목이 변경 내용을 정확히 반영하는가?
- [ ] 개요가 1-2문장으로 핵심을 전달하는가?
- [ ] 변경사항이 논리적 단위로 정리되었는가?
- [ ] 불필요한 섹션을 제거했는가?
- [ ] 민감 정보가 포함되지 않았는가?
